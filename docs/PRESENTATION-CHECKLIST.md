# คู่มือก่อนนำเสนอโปรเจกต์ RMU-Campus X

ใช้ checklist นี้ก่อนวันนำเสนอเพื่อลดโอกาส error ระหว่างสาธิต

---

## ก่อนวันนำเสนอ

### 1. ตรวจสอบ Build และ Type

```bash
npm run type-check   # ต้องผ่าน
npm run build        # ต้องผ่าน (ใช้เวลาประมาณ 1–2 นาที)
```

- ถ้า build ล้มเหลวจาก `.next/dev/types/validator.ts` ให้ลบโฟลเดอร์ `.next` แล้วรัน `npm run build` อีกครั้ง
- เราได้ตั้งค่าให้ `tsconfig` ไม่รวม `.next` แล้ว ดังนั้น `npm run type-check` จะไม่ขึ้นกับไฟล์ generate

### 2. ตรวจสอบ Environment (.env)

- ตรวจว่าไฟล์ `.env` มีค่าที่จำเป็นครบ (Firebase, Cloudinary, NEXT_PUBLIC_BASE_URL ฯลฯ)
- **NEXT_PUBLIC_BASE_URL** ต้องชี้ไปที่ URL ที่ใช้สาธิตจริง (เช่น `https://your-app.vercel.app` หรือ `http://localhost:3000`)

### 3. ข้อมูลทดสอบ

- เตรียมบัญชีผู้ใช้ 2 บัญชี (เจ้าของของ + ผู้ขอแลก) สำหรับสาธิต flow แลกเปลี่ยน
- ถ้ามี admin: เตรียมบัญชี admin สำหรับสาธิตหน้าจัดการ (ถ้าจะโชว์)

---

## วันนำเสนอ – ก่อนเริ่มสาธิต

### 4. รันแอปในโหมด Production (แนะนำ)

เพื่อให้สภาพใกล้เคียงของจริงและลดโอกาส error จากโหมด dev:

```bash
npm run build
npm run start
```

จากนั้นเปิดเบราว์เซอร์ที่ `http://localhost:3000` (หรือ port ที่แสดง)

- ถ้าสาธิตบน Vercel / hosting จริง: ตรวจว่า deploy ล่าสุดสำเร็จและเปิด URL ได้

### 5. เช็กเครือข่ายและบริการ

- **อินเทอร์เน็ต** – แอปใช้ Firebase, API ต่างๆ ต้องมีเน็ต
- **Firebase** – โปรเจกต์ยังเปิดใช้งานและ quota ไม่เกิน
- **LINE / Cloudinary** – ถ้าไม่สาธิตฟีเจอร์เหล่านี้ สามารถข้ามได้ แต่ไม่ควร error ถึงขั้นทำให้แอปค้าง (ควรมี error handling อยู่แล้ว)

### 6. เบราว์เซอร์

- ปิด DevTools (F12) ก่อนสาธิต หรือเตรียมแท็บที่ไม่มี Console เปิด เพื่อไม่ให้เห็น error ใน Console (ถ้ามี)
- ลองเปิดแอปในหน้าต่างแบบเต็มจอหรือ fullscreen เพื่อลดสิ่งรบกวน

## ลำดับสาธิตที่แนะนำ (ลดโอกาส error)

1. **หน้าแรก / ลงทะเบียนหรือล็อกอิน** – แสดงว่าผู้ใช้เข้าใช้งานได้
2. **Dashboard** – โหลดรายการของ
3. **โพสต์ของ** – สร้างรายการใหม่ (อัปโหลดรูปถ้ามี)
4. **ดูรายการ → กดขอแลก** – สร้าง exchange
5. **My Exchanges / แชท** – แสดงสถานะและแชท
6. **ยืนยันส่งมอบ/รับของ** – กดยืนยันจนสถานะเป็น “เสร็จสิ้น” (ถ้ามีเวลา)
7. **ฟีเจอร์เสริม** – Favorites, Notifications, Support ฯลฯ ตามเวลาที่มี

ถ้าเครือข่ายช้า: หลีกเลี่ยงการกดซ้ำหรือกดหลายปุ่มติดกัน รอ loading จบก่อนกดขั้นตอนถัดไป

---

## ถ้าเกิด Error ระหว่างสาธิต

- แอปมี **Error Boundary**: หน้า error จะแสดงข้อความแบบ “เกิดข้อผิดพลาด” พร้อมปุ่ม “ลองใหม่อีกครั้ง” และ “ย้อนกลับ” โดยไม่แสดง stack trace (ในโหมด production)
- แนะนำ: กด “ลองใหม่อีกครั้ง” หรือ “ย้อนกลับ” แล้วทำขั้นตอนเดิมอีกครั้งหนึ่ง
- ถ้าเป็นเรื่อง network: บอกกรรมการว่า “อาจเป็นเพราะการเชื่อมต่อ” และสาธิตส่วนอื่นที่ใช้ข้อมูลเดิมได้

## สรุปคำสั่งสำคัญ

| คำสั่ง | ใช้เมื่อ |
|--------|----------|
| `npm run type-check` | ตรวจว่าไม่มี TypeScript error |
| `npm run build` | สร้าง production build ก่อนสาธิต |
| `npm run start` | รันแอปแบบ production ท้องถิ่น |
| `npm run dev` | รันโหมดพัฒนา (ถ้าจำเป็น) |
